# Отраслевые кейсы использования

## Назначение документа

Этот документ показывает, в каких типах бизнеса и операционных контуров платформа особенно полезна. Он нужен не для описания внутреннего кода, а для понимания того, какие реальные задачи может решать система и почему ей нужен именно такой набор компонентов: registry, orchestration, Kafka, projections, monitoring, alerting, audit и Kubernetes-ready delivery.

## Финтех и платежные сценарии

### Где полезно

- обработка многошаговых внутренних workflow;
- автоматизация проверки и маршрутизации инцидентов;
- orchestration сценариев вокруг платежной инфраструктуры;
- эксплуатация процессов, где важны трассируемость, аудит и контроль стоимости.

### Почему платформа подходит

- execution run и execution steps дают понятную историю выполнения;
- audit trail помогает разбирать, кто запускал сценарий и что произошло;
- CQRS и event-driven архитектура удобны для интеграции с другими внутренними системами;
- anomaly detection и alerting помогают замечать деградацию до того, как она перерастет в инцидент.

### Пример

Команда платформы запускает сценарий анализа деградации платежного маршрута:

1. `planner` раскладывает задачу на шаги.
2. `tool_runner` собирает operational контекст.
3. `validator` дополнительно проверяет критичные выводы.
4. `reviewer` формирует итоговую рекомендацию.

Результат:

- прозрачный execution trail;
- контроль затрат;
- единая operational картина вместо ручного разбора через несколько систем.

## Электронная коммерция и маркетплейсы

### Где полезно

- автоматизация back-office сценариев;
- маршрутизация и проверка интеграционных процессов;
- оркестрация многошаговых процессов вокруг каталога, заказов и поддержки;
- эксплуатация distributed workflow с высокой изменчивостью нагрузки.

### Почему платформа подходит

- Kafka и worker-контур хорошо подходят для скачкообразной нагрузки;
- KEDA помогает масштабировать обработку по lag, а не только по CPU;
- registry-контур помогает централизованно управлять сценариями и их версиями;
- monitoring и cost tracking позволяют понимать, сколько стоит конкретный поток выполнения.

### Пример

Команда запускает сценарий обработки всплеска ошибок в order-processing pipeline:

- execution запускается для конкретного deployment;
- monitoring-контур собирает показатели latency и error rate;
- anomaly worker фиксирует всплеск;
- alert worker создает operational alert;
- команда быстро видит историю шагов и итоговую диагностику.

## Корпоративные платформенные команды

### Где полезно

- единая платформа для запуска и сопровождения внутренних сценариев;
- стандартизация delivery новых workflow;
- снижение числа “зоопарков” из разрозненных automation-сервисов;
- унификация observability и operational runbook.

### Почему платформа подходит

- модульная кодовая база позволила быстро стартовать без ранней избыточной сервисной дробности;
- сейчас эта же архитектура уже выведена в отдельные сервисы и процессы без смены базовых контрактов;
- Helm и Kubernetes-контур делают поставку воспроизводимой;
- role-based API и audit trail помогают встроить платформу во внутренние процессы доступа и контроля.

### Пример

Platform team использует платформу как общий execution backbone:

- одни команды регистрируют свои сценарии и deployment-ы;
- операторы смотрят executions, alerts и health;
- SRE используют Prometheus, OpenTelemetry и runbook;
- архитектурные границы остаются едиными для всей организации.

## Поддержка и incident response

### Где полезно

- запуск диагностических сценариев;
- обработка operational инцидентов;
- анализ цепочки событий и деградации;
- воспроизводимый response process для дежурной команды.

### Почему платформа подходит

- read-side API быстро показывает executions, anomalies, drift и alerts;
- structured logs, trace_id и correlation_id помогают связывать разные слои диагностики;
- health endpoints дают базовую и глубокую проверку зависимости;
- worker-based аналитика снимает часть ручной работы с дежурных инженеров.

### Пример

Во время инцидента оператор:

1. открывает `/api/v1/health/deep`;
2. проверяет `/api/v1/alerts` и `/api/v1/anomalies`;
3. находит нужный execution по `correlation_id`;
4. анализирует шаги выполнения;
5. принимает решение о повторном запуске или эскалации.

## Data/ML и model-serving контуры

### Где полезно

- эксплуатация сценариев, связанных с model endpoint-ами;
- контроль latency, token usage, стоимости и drift;
- единая operational прослойка вокруг inference pipeline.

### Почему платформа подходит

- orchestration-сервис публикует model inference events и cost events;
- drift worker отслеживает отклонения в метриках модели;
- cost monitoring помогает считать стоимость по workflow и окружению;
- registry хранит model endpoint-ы и версии в централизованном виде.

### Важное ограничение

Сейчас проект уже готов к эксплуатации сценариев вокруг моделей и inference endpoint-ов, но monitoring automation остается в основном rule-based. Это значит, что дрейф, аномалии и алерты уже есть, но полноценные reasoning-based monitoring agents пока не внедрены.

## Когда платформа особенно оправдана

Платформа особенно хорошо окупается, если у команды одновременно есть несколько признаков:

- много однотипных или похожих сценариев выполнения;
- нужен единый operational контур;
- важны аудит и трассируемость;
- есть Kafka или event-driven интеграции;
- нужна эксплуатация в Kubernetes;
- важно быстро масштабировать worker-ы по внешней нагрузке;
- команда не хочет собирать отдельный delivery и monitoring-контур для каждого нового сценария.

## Когда платформа может быть избыточной

Платформа может быть слишком тяжелой, если:

- нужен один простой synchronous API без фонового execution flow;
- нет требований к audit, observability и event-driven интеграциям;
- нет реальной потребности в worker-ах, Kafka и projections;
- сценарий полностью локальный и не требует production-grade эксплуатации.

В таком случае проект полезнее рассматривать не как “обязательный шаблон”, а как платформенный каркас для более зрелых и долгоживущих сценариев.
