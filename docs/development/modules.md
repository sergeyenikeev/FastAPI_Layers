# Справочник по модулям

## Назначение документа

Этот документ подробно описывает, из каких модулей состоит проект, что делает каждый модуль, где проходят его границы и зачем он нужен в общей архитектуре. Это практическая карта кодовой базы: ее удобно читать перед первым изменением, во время code review и при проектировании новых фич.

## Верхний уровень приложения

### `app/main.py`

Точка входа FastAPI-приложения.

Что делает:

- создает экземпляр FastAPI;
- подключает middleware, telemetry и маршруты;
- регистрирует lifecycle-хуки старта и остановки.

Зачем нужен:

- отделяет web-инициализацию от прикладной логики;
- держит вход в приложение тонким и предсказуемым;
- позволяет переиспользовать остальную часть системы из тестов и воркеров без запуска всего HTTP-слоя.

### `app/runtime.py`

Composition root приложения.

Что делает:

- собирает зависимости всего рантайма;
- выбирает publisher для тестового и обычного режима;
- создает command/query сервисы, health service, detector services, projection service и alerting service;
- управляет фоновыми задачами.

Зачем нужен:

- централизует сборку зависимостей;
- позволяет удобно подменять инфраструктурные зависимости в тестах;
- не дает разрастись “магии” инициализации по всему проекту.

### `app/api/`

HTTP-слой приложения.

Что делает:

- объединяет router-ы модулей;
- задает внешний API-контракт;
- связывает FastAPI-зависимости с runtime-сервисами.

Зачем нужен:

- отделяет transport layer от application services;
- удерживает API как тонкий слой оркестрации, а не место для бизнес-логики.

## Cross-cutting слой

### `app/core/`

Общий инфраструктурный слой, который нужен нескольким модулям сразу.

Что делает:

- `config.py` хранит типизированные настройки и чтение env vars;
- `security.py` реализует API key, JWT и RBAC-проверки;
- `middleware.py` управляет correlation id, trace id и HTTP-level обработкой запроса;
- `logging.py` и `observability.py` настраивают structured logging и tracing;
- `metrics.py` хранит Prometheus-метрики;
- `errors.py` задает общую модель доменных и прикладных ошибок;
- `context.py` дает доступ к correlation id и trace id внутри асинхронного выполнения.

Зачем нужен:

- собирает общую техническую инфраструктуру в одном месте;
- помогает модулям не дублировать cross-cutting код;
- удерживает согласованность логирования, трассировки и безопасности по всему приложению.

## Слой хранения и модели данных

### `app/db/`

Слой SQLAlchemy и доступа к PostgreSQL.

Что делает:

- `base.py` задает базовый ORM-класс и утилиты вроде `utc_now`;
- `session.py` создает engine и фабрики сессий;
- `models.py` описывает таблицы write/read side, processed events и operational сущности;
- `repositories.py` содержит тонкие repository-like функции там, где удобнее изолировать доступ к данным.

Зачем нужен:

- концентрирует работу с ORM и схемой базы;
- дает единое место для синхронизации моделей и миграций;
- упрощает ревью изменений в persistence-слое.

## Доменные контракты

### `app/domain/`

Общий язык системы на уровне DTO, событий и перечислений.

Что делает:

- `schemas.py` хранит общие Pydantic-схемы и DTO;
- `events.py` задает единый envelope для Kafka-событий и DLQ;
- `enums.py` хранит статусы и перечисления, используемые несколькими модулями.

Зачем нужен:

- фиксирует контракты между слоями;
- помогает не размазывать DTO по инфраструктурным модулям;
- делает event flow и API-контракты предсказуемыми.

## Messaging слой

### `app/messaging/`

Инфраструктурный слой транспорта событий.

Что делает:

- `topics.py` хранит карту топиков и DLQ-топиков;
- `kafka.py` реализует producer, consumer worker base class, сериализацию, idempotency, manual commit, retry и DLQ-публикацию.

Зачем нужен:

- отделяет event transport от прикладной логики;
- удерживает единые правила работы с Kafka;
- снижает риск расхождения поведения между разными worker-ами.

## Модуль `registry`

### `app/modules/registry/`

Контекст управления реестровыми сущностями платформы.

Что делает:

- принимает CRUD-команды по агентам, моделям, графам, deployment-ам, инструментам и окружениям;
- публикует доменные события в Kafka;
- отдает read-модели через query-side.

Основные файлы:

- `api.py` — HTTP-эндпоинты;
- `commands.py` — write-side use cases;
- `queries.py` — read-side запросы;
- `schemas.py` — модульные request/response-схемы.

Зачем нужен:

- отделяет управление каталогом сущностей от их выполнения;
- является основным write-side входом для конфигурации платформы;
- формирует факты, из которых затем материализуются read-модели.

## Модуль `orchestration`

### `app/modules/orchestration/`

Контекст исполнения сценариев.

Что делает:

- принимает запросы на запуск execution;
- собирает контекст deployment и model endpoint;
- запускает LangGraph workflow;
- публикует `execution.started`, `step.completed`, `execution.finished` и `execution.failed`;
- пишет cost и metric events по шагам исполнения.

Основные файлы:

- `graph.py` — описание `LangGraph`-workflow и его состояний;
- `service.py` — command-side запуск выполнения и публикация событий;
- `gateway.py` — адаптер обращения к внешнему model endpoint;
- `queries.py` — read-side доступ к execution run и execution steps;
- `schemas.py` — запросы и ответы orchestration API.

Зачем нужен:

- изолирует самый важный operational контур платформы;
- позволяет развивать workflow-логику отдельно от API и monitoring;
- уже сейчас готов к выносу в отдельный сервис при росте нагрузки.

## Модуль `monitoring`

### `app/modules/monitoring/`

Контекст health, performance, anomaly detection и drift detection.

Что делает:

- отдает `/health/live`, `/health/ready`, `/health/deep`;
- агрегирует read-модели метрик, стоимости, аномалий и дрейфа;
- применяет детекторы threshold, rolling mean/std, z-score, PSI и divergence-метрики.

Основные файлы:

- `health.py` — проверки зависимостей и worker-состояния;
- `queries.py` — read-side отчеты и выборки;
- `anomaly.py` — pluggable anomaly detectors;
- `drift.py` — pluggable drift detectors;
- `api.py` — monitoring endpoints;
- `schemas.py` — DTO мониторинга.

Зачем нужен:

- держит эксплуатационные сигналы рядом и в одном языке;
- позволяет развивать observability и analytics без смешивания с execution-логикой;
- делает аномалии и дрейф самостоятельными модулями, а не побочным кодом вокруг метрик.

## Модуль `alerting`

### `app/modules/alerting/`

Контекст генерации и отправки алертов.

Что делает:

- создает alert-сущности;
- применяет dedupe и cooldown;
- отправляет уведомления через webhook, email stub и логирование;
- отдает read-side для списка и просмотра алертов.

Основные файлы:

- `service.py` — логика dedupe, cooldown и dispatch;
- `queries.py` — read-side выборки;
- `api.py` и `schemas.py` — контракт внешнего API.

Зачем нужен:

- защищает операторов от alert storm;
- отделяет аналитику сигналов от механики их доставки;
- упрощает дальнейшее добавление реальных каналов уведомлений.

## Модуль `audit`

### `app/modules/audit/`

Контекст трассируемости действий.

Что делает:

- публикует audit events на write-side;
- отдает историю audit trail через read-side;
- хранит correlation id и trace id рядом с действием.

Зачем нужен:

- делает изменения и действия наблюдаемыми;
- упрощает расследование инцидентов и проверку доступа;
- поддерживает secure-by-default подход.

## Projection слой

### `app/projections/`

Слой материализации read-моделей из событий.

Что делает:

- принимает event envelope;
- маршрутизирует его в нужный handler по `event_type`;
- обновляет PostgreSQL read-модели через upsert/update/delete;
- переживает out-of-order события и безопасно создает placeholder-сущности там, где это необходимо.

Зачем нужен:

- связывает Kafka write-path с API read-path;
- удерживает read side идемпотентным и переигрываемым;
- делает CQRS практичным, а не декларативным.

## Workers

### `app/workers.py` и `app/worker.py`

Точка сборки фоновых consumer-ролей.

Что делает:

- задает, какие worker-ы запускаются для projection, analytics и alerts;
- привязывает handler-ы к consumer group и Kafka topics;
- позволяет запускать роли по переменной окружения.

Зачем нужен:

- отделяет фоновые процессы от HTTP-рантайма;
- дает простой путь к масштабированию через Kubernetes, KEDA и разные deployment-ы;
- удерживает карту background processing в одном месте.

## Инфраструктура и операционный слой

### `scripts/`

Служебные утилиты для локальной разработки и диагностики.

Что делает:

- `dev_stack.py` поднимает и останавливает локальный стек;
- `kafka_debug.py` помогает смотреть топики, consumer groups, lag и payload событий;
- `create_topics.py` подготавливает Kafka topics.

Зачем нужен:

- снижает стоимость локального старта;
- упрощает эксплуатационную диагностику без ручных `docker exec`;
- помогает разработчику воспроизводить production-потоки локально.

### `tests/`

Слой автоматической проверки системы.

Что делает:

- `tests/unit/` проверяет чистую прикладную логику;
- `tests/integration/` проверяет API, projections, event flow и runtime-интеграции;
- `conftest.py` собирает тестовые фикстуры и подмены рантайма.

Зачем нужен:

- страхует рефакторинг;
- подтверждает согласованность между write-side, Kafka и read-side;
- удерживает систему пригодной для эволюции.

### `alembic/`, `docker/`, `deploy/`, `helm/`

Инфраструктурный контур поставки и эксплуатации.

Что делает:

- `alembic/` управляет миграциями схемы базы;
- `docker/` описывает сборку контейнеров;
- `deploy/` хранит дополнительные deployment-примеры;
- `helm/` описывает production-ready поставку в Kubernetes.

Зачем нужен:

- связывает исходный код с реальной эксплуатацией;
- не дает документации и манифестам жить отдельно от кода;
- позволяет рассматривать репозиторий как целостный delivery-артефакт.

## Как использовать этот справочник

- Если меняется один модуль, сначала найдите его раздел в этом документе и проверьте соседние слои, которые он затрагивает.
- Если задача затрагивает событие, почти всегда нужно смотреть вместе `commands/service -> messaging -> projections -> queries/api`.
- Если задача затрагивает внешний интеграционный контракт, дополнительно проверьте `docs/`, `tests/`, `scripts/` и Helm-конфиги.
