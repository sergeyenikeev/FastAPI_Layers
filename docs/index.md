# Платформа эксплуатации сценариев

Централизованная платформа эксплуатации распределенных сценариев со следующими ключевыми компонентами:

- управляющая плоскость на `FastAPI`
- движок выполнения сценариев
- событийная магистраль на Kafka с DLQ и идемпотентными потребителями
- проекции в `PostgreSQL` для CQRS-моделей чтения
- `Redis` для координации и механизмов heartbeat
- `Prometheus`, `OpenTelemetry` и структурированные журналы
- артефакты для Kubernetes и Helm с поддержкой `KEDA`

Платформа следует стратегии модульного монолита, чтобы каждый ограниченный модуль можно было позже вынести в отдельный сервис без изменения контрактов событий.

## Бизнес-описание проекта

Платформа решает прикладную задачу промышленной эксплуатации распределенных сценариев и многошаговых автоматизаций. Ее смысл не в том, чтобы просто “запустить один сервис”, а в том, чтобы дать команде единое место для регистрации исполняемых сущностей, описания сценариев, подключения моделей и инструментов, запуска выполнений, наблюдения за качеством работы и контроля операционных рисков.

Что платформа делает для бизнеса:

- ускоряет вывод новых сценариев в эксплуатацию за счет готового registry-контура, API, Docker, Helm и Kubernetes-артефактов;
- снижает стоимость поддержки за счет централизованных health-проверок, мониторинга, аудита, алертинга и event-driven архитектуры;
- делает эксплуатацию прозрачной: можно понять, какой сценарий запущен, что происходило на каждом шаге, сколько это стоило и где возникли отклонения;
- помогает стандартизировать delivery новых сценариев внутри команды или нескольких команд;
- уменьшает риск “теневых” интеграций, когда каждый новый workflow живет отдельно и без общей observability-модели.

Где платформа полезна:

- в платформенных командах, которые хотят стандартизировать запуск и сопровождение сценариев;
- в продуктовых командах, которым нужен единый контур для многошаговых workflow, мониторинга и аудита;
- в enterprise-среде, где важно не только выполнить сценарий, но и доказуемо показать историю изменений, запусков, ошибок и стоимости;
- в средах с несколькими окружениями, где deployment должен быть воспроизводимым и управляемым через Kubernetes и Helm;
- в сценариях, где важна постепенная эволюция архитектуры: сначала modular monolith, позже возможный split по сервисам без смены базовых контрактов.

### Главная ценность платформы

Если коротко, проект дает не “один workflow”, а эксплуатационный каркас вокруг workflow:

- реестр сущностей;
- запуск выполнений;
- событийную магистраль;
- read-side проекции;
- мониторинг и диагностику;
- аудит и алертинг;
- готовый путь поставки в Kubernetes.

Именно эта комбинация делает проект полезным для production, а не только для локального прототипирования.

## Пользовательский сценарий использования

Ниже приведен типовой путь, по которому платформу использует инженер, оператор или platform team.

### Сценарий 1. Подготовка среды

Сначала пользователь или команда платформы:

- регистрирует окружения через `/api/v1/environments`;
- регистрирует model endpoint-ы через `/api/v1/models`;
- регистрирует инструменты через `/api/v1/tools`;
- создает graph definition через `/api/v1/graphs`;
- создает агента и его версию через `/api/v1/agents`.

Результат этого этапа: в системе появляется управляемый каталог сущностей, пригодный для последующего деплоя и запуска.

### Сценарий 2. Подготовка исполняемой конфигурации

Далее создается deployment через `/api/v1/deployments`.

Deployment связывает между собой:

- агента;
- граф выполнения;
- модель;
- окружение;
- execution-конфигурацию.

Именно deployment становится точкой, на которую потом можно ссылаться при запуске выполнений.

### Сценарий 3. Запуск выполнения

Оператор или внешний сервис вызывает `/api/v1/executions` и передает:

- `deployment_id`, если нужно запустить уже подготовленную конфигурацию;
- либо `graph_definition_id`, если нужно запустить сценарий напрямую;
- `input_payload`, описывающий задачу и входной контекст.

После этого происходит следующий путь:

1. API принимает команду.
2. Система публикует `execution.started` в Kafka.
3. Orchestration-слой запускает workflow.
4. Каждый шаг публикует `step.completed`.
5. В финале публикуется `execution.finished` или `execution.failed`.
6. Projection worker материализует read-side состояние в PostgreSQL.
7. Пользователь видит результат через API чтения.

### Сценарий 4. Наблюдение за выполнением

После запуска пользователь или оператор может:

- посмотреть список запусков через `/api/v1/executions`;
- открыть конкретное выполнение и увидеть шаги;
- посмотреть health-состояние через `/api/v1/health/*`;
- посмотреть операционные метрики через `/api/v1/metrics` и `/api/v1/metrics/summary`;
- посмотреть стоимость через `/api/v1/costs`;
- посмотреть аномалии через `/api/v1/anomalies`;
- посмотреть дрейф через `/api/v1/drift`;
- посмотреть алерты через `/api/v1/alerts`;
- посмотреть аудит через `/api/v1/audit`.

### Сценарий 5. Эксплуатация в Kubernetes

Для production-поставки команда использует Helm chart и Kubernetes-манифесты:

- API и worker-ы деплоятся отдельно;
- readiness и liveness используют health-ручки;
- Prometheus собирает `/metrics`;
- KEDA масштабирует worker-ы по lag в Kafka;
- Ingress отдает внешний доступ;
- Secret и External Secret-подход управляют чувствительными значениями.

### Кто обычно является пользователем платформы

- platform engineer, который управляет инфраструктурной частью;
- backend engineer, который добавляет новые сценарии и интеграции;
- operator или SRE, который следит за инцидентами и выполнениями;
- product team, которая использует платформу как execution backbone для своих бизнес-процессов.
